5. Veri ALMA (Backoffice → App)

AkışAçıklama

İki paralel yol: App veriyi Local Backend veya Cloud’dan alabilir; önce Local denenir.

5a. Zorunlu Tetikleyici (Force Update)

Cloud’da yapılan fiyat/katalog değişikliğinin tabletlere anında yansıması için periyodik sync beklenmez. Backoffice değişiklik kaydettiğinde Cloud, cihazlara "hemen güncelle" sinyali gönderir.

YöntemAçıklama
Push (FCM)Backoffice kaydedince Cloud, FCM ile "catalog_updated" gönderir → App hemen sync başlatır
Manuel butonBackoffice’de "Zorunlu Güncelle" butonu → Cloud API çağrısı → Cihazlar bir sonraki kısa polling’de sync alır

Sonuç: Yeni fiyatlar periyodik sync (5–15 dk) beklenmeden anında veya kısa sürede tabletlere yansır.

8. App'ler Neden Doğrudan Haberleşmesin?

App'ler teknik olarak doğrudan (P2P) haberleşebilir. Ancak 3+ cihaz, tutarlılık ve conflict yönetimi için Backend üzerinden haberleşmek daha uygundur.

MimariNe zaman?
App ↔ App (doğrudan)2–3 cihaz, basit senaryo; conflict zor
App ↔ Backend ↔ App3+ cihaz, tek merkez, conflict kolay ✓
HibritBackend merkez + sadece "veri değişti" uyarısı için App↔App (isteğe bağlı)

Karar: Bu mimaride App'ler Backend üzerinden haberleşir; doğrudan App↔App yok.

9. Özet Kurallar

  • Local Backend = Yerel merkez (App + KDS buradan beslenir)
  • Cloud = Backoffice merkezi
  • 3 seviye failover: Local A → Local B → Cloud
  • Yukarı (veri gönderme): App + Local Backend → Cloud (audit için)
  • Aşağı (veri alma): Backoffice → Cloud → Local Backend + App (iki yol, fallback)
  • Zorunlu Tetikleyici (Force Update): Fiyat/katalog değişince Cloud, cihazlara anında güncelleme sinyali gönderir (FCM veya manuel)
  • Gün sonu: Cloud, App vs Local verilerini karşılaştırır, fark raporu üretir
  • Kaynak olarak: Local Backend (gün içi veri için)
  • App haberleşmesi: App ↔ App doğrudan yok; hepsi Backend üzerinden
